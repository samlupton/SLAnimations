# ConfettiAnimation Project Documentation

## Overview

This repository contains a single Swift package product: `SharedConfetti`.

The package provides:

- A typed configuration model under the `Confetti` namespace.
- A UIKit renderer built on `CAEmitterLayer` and `CAEmitterCell`.
- A small SwiftUI bridge for embedding the renderer in SwiftUI layouts.
- A convenience helper for emitting confetti over the app's top-most window.

The current implementation is intentionally compact. Most behavior is driven by static configuration factories rather than a large public API surface.

## High-Level Architecture

The codebase is organized around four layers:

1. `Confetti` namespace and nested value types
   Defines the data used to describe emitter cells, including motion, scaling, lifetime, and visual content.
2. Configuration factories
   Build ready-to-use presets such as `shower` and `fountain` by combining assets and cell settings.
3. Rendering layer
   Converts typed `Confetti.Cell` values into `CAEmitterCell` instances and attaches them to a `CAEmitterLayer`.
4. Integration layer
   Exposes UIKit and SwiftUI entry points, plus a helper to overlay confetti across the active window.

## Runtime Flow

The active rendering path is:

1. A caller chooses a `Confetti.Configuration`, such as `.shower` or `.fountain`.
2. A `ConfettiView` is created with that configuration.
3. Calling `emit()` asks the view to:
   - build `CAEmitterCell` objects from `configuration.cells`
   - build a configured `CAEmitterLayer`
   - add the emitter as a sublayer of the view
   - optionally attach a `CABasicAnimation`
4. The `ConfettiRenderer` protocol extension copies each typed field from `Confetti.Cell` into the Core Animation emitter cell.

`emitConfetti(configuration:)` is the convenience entry point that performs this flow by installing a `ConfettiView` into the top active window.

## Models

### `Models/Confetti/Confetti.swift`

Declares:

- `public enum Confetti {}`

This empty enum is used as a namespace. All public model types are nested under it, which keeps the API surface grouped and avoids polluting the module with many top-level types.

### `Models/Confetti/Extensions/Confetti+Cell/Confetti+Cell.swift`

Defines the main particle model:

- `Confetti.Cell`

Properties:

- `lifetime`
- `spin`
- `scale`
- `acceleration`
- `velocity`
- `emission`
- `contents`

These fields mirror the properties later copied onto `CAEmitterCell`.

The file also contains internal factories:

- `makeShowerCells(with:)`
- `makeFountainCells(with:)`

These create arrays of `Confetti.Cell` using hard-coded preset physics values:

- `shower` uses a full 360-degree emission spread, moderate velocity, downward acceleration, and shrinking scale over time.
- `fountain` uses an upward launch angle, higher birth rate, stronger upward-style emission intent, and smaller scale.

### `Confetti+Acceleration.swift`

Defines `Confetti.Cell.Acceleration`:

- `x`
- `y`
- `z`

This maps directly to `CAEmitterCell` acceleration components.

### `Confetti+Emission.swift`

Defines `Confetti.Cell.Emission`:

- `latitude`
- `longitude`
- `range`

This controls the direction and spread of particle emission.

### `Confetti+Lifetime.swift`

Defines `Confetti.Cell.Lifetime`:

- `birthRate`
- `base`
- `range`

These values map onto particle generation rate and lifetime randomness.

### `Confetti+Scale.swift`

Defines `Confetti.Cell.Scale`:

- `base`
- `range`
- `speed`

This controls particle size, variation, and scaling animation over the particle lifetime.

### `Confetti+Spin.swift`

Defines `Confetti.Cell.Spin`:

- `base`
- `range`

This controls angular velocity and angular variance.

### `Confetti+Velocity.swift`

Defines `Confetti.Cell.Velocity`:

- `base`
- `range`

This controls the starting speed of each emitted particle.

### `Confetti+Content.swift`

Defines `Confetti.Cell.Contents`:

- `image`
- `rect`
- `scale`

This type describes the visual payload of a particle. The internal factory methods:

- `makeShowerContents()`
- `makeFountainContents()`

load `CGImage` instances from package assets and wrap them as `Contents`.

### `Confetti+Assets.swift`

Defines the internal asset loader:

- `Confetti.Assets.makeShowerAssets()`
- `Confetti.Assets.makeFountainAssets()`

The code expects generated asset resource symbols for:

- `i1` through `i8`
- `water1` through `water4`

Those are the visual particle images used by the presets.

### `Confetti+Configuration.swift`

Defines the top-level configuration object:

- `Confetti.Configuration`

It currently stores only:

- `cells: [Cell]`

Public preset entry points:

- `Confetti.Configuration.shower`
- `Confetti.Configuration.fountain`

Internal factory methods:

- `makeShower(with:)`
- `makeFountain(with:)`

This is the actual preset system in the current codebase. Despite the `Presets` folder name elsewhere, these factories are the active preset implementation.

## Rendering

### `Protocols/ConfettiRenderer.swift`

Defines the internal rendering contract used by `ConfettiView`.

Required methods:

- `makeConfetti(with:in:)`
- `makeCells()`
- `makeAnimation()`
- `makeCACell(_:cell:)`
- `makeAcceleration(_:cell:)`
- `makeContent(_:cell:)`
- `makeEmission(_:cell:)`
- `makeScale(_:cell:)`
- `makeVelocity(_:cell:)`
- `makeLifetime(_:cell:)`
- `makeSpin(_:cell:)`

The extension provides the default implementation for everything except the top-level emitter creation and optional animation decision.

Important implementation detail:

- `makeCACell` is a composition method that delegates to each smaller property-mapping function.
- The protocol extension is where the typed Swift model is translated into Core Animation emitter state.

One nuance in `makeContent`:

- It sets `cacell.scale = cell.contents.scale`

and then `makeScale` later sets:

- `cacell.scale = cell.scale.base`

So `contents.scale` is effectively overwritten by `scale.base` in the current mapping order. That means `Contents.scale` is presently not an independent runtime control.

### `Presentation/ConfettiView.swift`

Defines the main UIKit renderer:

- `public final class ConfettiView: UIView, @MainActor ConfettiRenderer`

Responsibilities:

- Store a `Confetti.Configuration`
- Build `CAEmitterCell` objects from `Confetti.Cell`
- Build a `CAEmitterLayer`
- Add that emitter to its own backing layer
- Attach an optional animation

Key methods:

- `init(configuration:)`
- `emit()`
- `makeConfetti(with:in:)`
- `makeCells()`
- `makeAnimation()`

Emitter behavior in `makeConfetti(with:in:)`:

- Birth rate: `1`
- Layer lifetime: `1`
- Position: top-center of the view
- Size: full view width, zero height
- Shape: `.line`
- Mode: `.outline`
- Opacity: `0.5`

Implications:

- The current emitter geometry is tailored to a top-down shower effect.
- The `fountain` preset changes cell physics, but not emitter geometry.
- Because `makeAnimation()` returns `nil`, emission does not currently include a burst/fade animation stage.

### `Presentation/ConfettiViewRepresentable.swift`

Defines:

- `public struct ConfettiViewRepresentable: UIViewRepresentable`

Responsibilities:

- Instantiate `ConfettiView` for SwiftUI
- Provide an empty `updateUIView`

Important limitation:

- It always creates `ConfettiView()` using the default `.shower` configuration.
- It does not expose configuration injection.
- It does not trigger `emit()` automatically.

The doc comment in the file explicitly states that the caller is responsible for triggering emission.

## Convenience API

### `Utils/Helper.swift`

Contains two pieces:

1. `UIWindow.getTopWindow(at:)`
2. `@MainActor public func emitConfetti(configuration:)`

`getTopWindow(at:)`:

- Filters connected scenes to foreground-active `UIWindowScene` instances.
- Flattens windows across those scenes.
- Returns the last window that is either the key window or matches the requested level.

`emitConfetti(configuration:)`:

- Finds the active window.
- Creates a `ConfettiView` with the given configuration.
- Sizes it to the window bounds.
- Disables user interaction.
- Adds it as a subview.
- Calls `emit()`.

This is the simplest public integration point for app-wide celebration effects.

Important operational note:

- The helper adds a view but does not remove it afterward.
- Since `makeAnimation()` is `nil`, there is no explicit cleanup behavior in the current implementation.

## Legacy / Inactive Preset Files

The `Presets` folder contains commented-out historical implementations rather than active source.

### `Presets/Shower.swift`

Old prototype for a `Shower` type conforming to a `ConfettiGenerator` protocol. The entire file is commented out.

It shows an earlier design where:

- assets were passed in as `UIImage`
- emitter logic lived in a per-preset type
- cell properties were configured directly on `CAEmitterCell`

This file is useful as historical context, but it does not participate in the build.

### `Presets/Fountain.swift`

Commented-out prototype of a `Fountain` generator. Also not part of the active implementation.

It demonstrates an earlier emitter geometry that launched particles from the bottom-center using `.point` emission.

That behavior is notable because the current active renderer in `ConfettiView` does not replicate this emitter geometry.

### `Presets/LeftCannon.swift`

Commented-out prototype of a left-side cannon effect with:

- point emission from the left edge
- a short birth-rate animation
- direct emitter cell setup

This is inactive and serves as a hint that the project may eventually support more directional presets than it currently exposes.

## Design Notes and Tradeoffs

### What the package does well

- Uses typed configuration models instead of exposing raw `CAEmitterCell` mutation everywhere.
- Keeps the active rendering code small and easy to reason about.
- Offers both UIKit and SwiftUI integration paths.
- Keeps preset creation centralized in configuration factories.

### Current limitations

- `ConfettiViewRepresentable` is not configurable.
- `makeAnimation()` always returns `nil`, so there is no built-in burst taper or teardown animation.
- `emitConfetti(configuration:)` does not remove the overlay view after emission.
- The active emitter geometry is fixed to a top line, which means `.fountain` only changes particle physics, not origin geometry.
- `Contents.scale` is overwritten by `Scale.base` during renderer mapping.
- The `Presets` directory is inactive and may confuse readers unless documented clearly.
- The test target is still a scaffold.

## Suggested Mental Model For Future Work

If you need to extend this library, the cleanest current path is:

1. Add or refine `Confetti.Cell` supporting types if new particle parameters are needed.
2. Add a new `Confetti.Configuration` factory for a new preset.
3. If the preset needs different emitter geometry, make `ConfettiView` configurable at the emitter-layer level instead of only the cell level.
4. Add removal or completion handling to the helper overlay path.
5. Replace placeholder tests with assertions around configuration generation and renderer mapping.
